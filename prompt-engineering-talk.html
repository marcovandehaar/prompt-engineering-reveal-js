What is Prompt Engineering?

Prompt engineering is the practice of crafting and optimizing inputs to AI models to get more accurate, relevant, and useful outputs.

Prompt engineering is the process of crafting and refining input instructions or queries to guide AI systems, like ChatGPT, toward producing the most accurate, relevant, or creative responses.

Missing Typical Constructs

You can‚Äôt build skyscrapers out of sticky notes
  • ‚ùå No functions: Can‚Äôt extract logic into reusable named steps
  • ‚ùå No reusable variables: Can‚Äôt store intermediate results and reference them later like code

So we‚Äôve seen how even a carefully worded prompt can fall apart under scrutiny ‚Äî and this next point shows just how deeply that limitation goes.

Programming languages give us powerful constructs: functions, variables, conditionals. These aren‚Äôt just nice to have ‚Äî they‚Äôre what let us build complex, reliable systems by composing smaller pieces of logic.

But in prompt engineering, that composability is missing. You can‚Äôt extract a reusable piece of logic and apply it elsewhere. There‚Äôs no way to define a named function, pass in arguments, or return structured values.

You also can‚Äôt store and reuse intermediate results ‚Äî there's no state, no scope. The model might "remember" what you just said, or it might reinterpret it based on new context.

That‚Äôs what the sticky note metaphor is about. Each prompt feels self-contained, disconnected. You can write a brilliant one ‚Äî but you can‚Äôt build on it. And without modularity, there‚Äôs no real foundation for building scalable systems.

Let‚Äôs close by coming back to the core reasons prompt engineering doesn‚Äôt meet the definition of a programming language.

First, it lacks a formal specification. Programming languages are defined by strict syntax and behavior ‚Äî through grammars, type systems, and execution models. Prompting has none of that. There‚Äôs no agreed-upon structure, no guaranteed behavior, and no enforcement.

Second, prompts are entirely dependent on model implementation. In programming, code compiles the same way on every standards-compliant compiler. But a prompt that works well in GPT-4 might fall apart in Claude or Gemini. It‚Äôs not a language ‚Äî it‚Äôs a conversation tuned to a specific listener.

And finally, natural language itself is ambiguous. That‚Äôs a feature when we talk to people ‚Äî but a serious limitation when trying to control a system. We‚Äôve seen that even well-phrased prompts can be misunderstood, reinterpreted, or produce subtly different results each time.

Key Evidence Against

  • Inconsistent results across models
  • No compiler or formal verification
  • Rapid changes in best practices
  • Limited debugging capabilities
So while the case for the motion paints a hopeful picture, these points are why we need to be careful before calling prompt engineering a programming language.

First ‚Äî the inconsistency. A true programming language executes predictably across environments. Prompts, on the other hand, can vary wildly depending on the model or provider ‚Äî and even across runs.

There‚Äôs no compiler. No way to formally validate a prompt before ‚Äúrunning‚Äù it. We can‚Äôt catch errors, only observe undesired behavior after the fact.

Best practices in prompting change rapidly ‚Äî not because we‚Äôre refining a language specification, but because we‚Äôre trying to reverse-engineer what works.

And lastly ‚Äî debugging. In code, we isolate the issue, step through logic, and fix it with confidence. In prompting, we revise, rephrase, and hope it works better next time.

These aren‚Äôt just quirks ‚Äî they‚Äôre fundamental differences. They show that while prompt engineering is a valuable skill, it doesn‚Äôt yet meet the bar of a true programming language.
